# MindWP repository audit and improvement proposal — 22 July 2026

Status: **audit and proposal only.** Nothing in this document has been applied. Every recommendation waits for explicit approval. This document is a dated audit record, not a canonical authority; the owners named per finding remain the authorities.

---

## 1. Executive verdict

**Ready after minor corrections.**

The documentation system, skills, CSS foundations, tooling and test suite are genuinely strong — recently consolidated, internally consistent, and unusually honest about past failure modes. No foundational rework is needed. What stands between the repository and substantial new page work is a short list of small, precise corrections (a wrong global typography role, dead footer links, a page-plans file with no declared status, three stale references, one mojibake line) plus one **large but already-planned execution**: the Homepage still implements the superseded "Work comes in. Too much slips away." thesis while STRATEGY.md and PAGE-PLANS.md define a website-first replacement. That is page execution, not foundation repair — and the repository's own rules correctly frame it as reportable drift, which this audit now formally reports.

**Design language: preserve and clarify. Do not redefine.**

Rendered evidence shows a recognisable, genuinely premium language — two-tone Fraunces headlines, hairline geometry, navy/paper/mist surfaces with emerald discipline, constructed-artifact illustration, meaning-bearing diagrams — that is stronger than the "generic agency output" fear suggests. The weaknesses found are execution-era artifacts (an empty proof section, near-blanket entrance reveals, one repeated section anatomy) and one foundation defect (the global `h3` role that both implemented pages fight), not defects of the language itself. The language should be preserved, its few genuinely shared signatures named, and one typography role corrected.

## 3. What is already strong

These are foundations to keep, and they are the majority of the system:

- **FOUNDATION.md** — a genuinely durable business-truth document. It already contains the decision rule this audit needed: website-led is an offer hierarchy; a leakage framing "should not become the organising brand thesis"; memorable copy is not sufficient reason to reverse the hierarchy (line 15).
- **STRATEGY.md** — clear offer architecture (Smart Website Systems principal; four optional supporting services with named boundaries), an explicit Homepage positioning consequence, honest open-decisions register, and a defined "useful enquiry" boundary between SWS and Lead Response. This is better strategy hygiene than most real agencies maintain.
- **WRITING.md** — concrete voice and claims discipline; the final-copy checklist is directly usable.
- **PAGE-PLANNING.md** — the compact page-brief contract (fixed / adaptable / open / unavailable; blocking vs deferrable) is exactly the right planning-vs-design boundary. "A plan row is meaning material… not automatically one horizontal visual band" addresses the historical failure directly.
- **DESIGN.md** — an unusually mature design authority: three design layers, the meaning→composition bridge, compositional variables, four doctrine examples, recursive page/focal-section method, anti-generic questions as judgement prompts (not quotas), and careful accepted-page evidence rules. It reads as distilled scar tissue from precisely the problems in this audit's brief.
- **ENGINEERING.md** — correct ownership tables (CSS owner ladder, validation-by-risk), honest command inventory, publication boundaries.
- **The two-skill system** — `mindwp-design-build` and `mindwp-frontend-quality` are well-scoped, non-overlapping, load bounded context, and protect requested scope. The earlier four-skill workflow was rightly retired.
- **Tooling** — `export-context` (explicit, flag-driven, no inference), `capture-route` (viewport matrix, reduced-motion, section crops, refuses in-repo output), `check-skills` (structural validation), all with Node test coverage. CI runs check + build + Playwright.
- **Test suite** — keyboard navigation for all three Homepage tab systems, mobile orientation switching, overflow guards at 400/360, reduced-motion content-visibility assertions, scroll-scrub progress, and publication-state tests (noindex + sitemap absence for the unreleased Local SEO route). This is real engineering discipline.
- **Global CSS foundation** — ~690 lines total, semantic tokens with aliases, an elegant font-shorthand type scale (`--display-*`/`--body-*` applied via `font:`), one easing family, restrained shadows, a single `.section`/`.container` grammar. Small, learnable, and heavily used by both pages (139 spacing-token uses in `home.css` alone).
- **Local SEO Authority page** — the strongest current evidence of the design language working *with* the current strategy: "Website — the foundation" expressed spatially (cards standing on a navy slab), honest "Illustrative view / Illustrative framework" labels, quiet/dominant variation inside one stack, rotated overlapping promise cards, giant emerald display numerals.
- **Publication discipline** — `LIVE_ROUTES` contains only `/`; the Local SEO page is built but `noindex` and absent from navigation/sitemap, with tests enforcing that state.

## 6. Writing and page-planning audit

**Writing:** WRITING.md is discoverable, specific and matches the implemented voice (the rendered pages demonstrate its cadence: short declaratives, concrete objects, one thought per sentence). The implemented pages contain two writing-level compliance issues, both owned by the old-era Homepage rather than the authority: the inspectability claim over placeholder work ("Website work you can inspect" — nothing is inspectable), and unapproved founder-experience claims (§5). No changes to WRITING.md recommended.

**Page planning:** PAGE-PLANNING.md is strong; the gap is between it and `PAGE-PLANS.md`:

- **Status is undeclared.** PAGE-PLANS.md is not in the docs/README.md authority table, its H1 calls it "page strategy," and nothing states whether its ~20 per-page sections are approved briefs, planning reservoirs, or drafts. A fresh session told to "build the Smart Website Systems page" cannot tell what it may adapt.
- **The rows carry none of the contract fields.** Each section is `# | Eyebrow | Heading` only — no essential meaning, no proof inventory, no boundaries, no fixed/adaptable/open/unavailable marks, no unresolved-decisions list. PAGE-PLANNING.md explicitly requires those to prevent both invention and 16-band literalism; a heading-only table invites exactly the historical "every row becomes a horizontal section, headings become final copy" failure. (The implemented Homepage grouped its 16 planned rows into 15 sections — near 1:1.)
- **Sequence and depth are good.** Page jobs are crisp, SEO intents are researched and plausible, industry pages are genuinely differentiated by decision (not name-swapped), and truth boundaries appear in the page jobs. The raw material is high quality; it is the *contract metadata* that is missing.

**Smallest correction (no new workflow):** (1) add PAGE-PLANS.md to the docs/README.md table with an explicit role — "catalogue of page jobs, SEO intents and working section meanings; not yet execution briefs"; (2) add a five-line preamble to PAGE-PLANS.md stating that headings are working planning language (adaptable unless marked), section rows are meaning material not bands, and that before execution a page's entry must be promoted to a brief carrying the PAGE-PLANNING.md status fields (fixed/adaptable/open/unavailable + proof + unresolved decisions). No new documents, no new skill.

## 7. Current MindWP design-language definition (reconstructed from evidence)

**Brand character.** Considered specialist, editorial-operational hybrid. Calm, dark-anchored premium: conviction carried by navy rooms and large serif statements; explanation carried by light rooms and quiet sans detail. Confidence expressed through restraint and precision (hairlines, tight tracking, controlled emerald) rather than decoration. No urgency theatre anywhere — even the "leaks" argument is delivered in measured, aphoristic prose.

**Typography (the core of the identity).**
- Fraunces for conviction: H1 up to ~6.5–6.7rem, tight leading (0.96–1.02), negative tracking at hero scale; Inter for operation.
- **The two-tone turn**: nearly every major heading is a two-part rhetorical structure — bold roman statement + muted *italic* turn ("Work comes in." / *Too much slips away.*; "Reviews shorten *the distance to trust.*"; "A local presence needs tending, *not a launch date.*"). Implemented divergently: `.home-heading-muted` span on Home, `em` + `--lseo-muted-display` on Local SEO.
- Rhetorical typographic marks as design material: strikethrough vendor list, underlined promise words, giant emerald Fraunces numerals (01–05), italic serif editorial asides (`.home-editorial-note`), caps eyebrows with a preceding em-dash rule (built into `.eyebrow::before`).
- Fraunces variation axes (`--display-settings`: SOFT 20, WONK 1, opsz 144) — defined globally, used exactly once (Local SEO hero), where it produces the system's best typographic moment.

**Colour and surfaces.** Navy (#071629) with panel/raised/hairline ladder; paper and mist for light rooms; white cards; emerald strictly as attention (eyebrows, marks, connection lines, active states, on-dark CTAs) with deep-emerald on light. Semantic roles hold in practice — raw hex in page CSS is rare (~8 values on Home, mostly the work-card tints; ~7 greys on LSEO). Warning/danger exist but barely appear.

**Composition and space.** One `.container` system (max 1440) with narrow/content/wide variants; generous vertical rhythm (clamp 4–6rem); split intros, centred statements, and asymmetric splits; layering used sparingly but meaningfully (floating chips over panels, cards standing on a foundation slab, one rotated card fan). Diagrams are geometric and sparse: radial rings, hub-and-spoke fans, connector-line stacks, horizontal rule timelines.

**Geometry and signatures.** Hairline discipline is the strongest geometric signature — 1px rules structure lists, steps, footers, and separations everywhere. Restrained radii (0.5/0.75rem), rare soft shadows, ghosted translucent panels on navy. The eyebrow-dash, the hairline list, the two-tone turn, the emerald connection line, and constructed mini-artifacts are the recurring identity elements.

**Visual material.** Entirely constructed: mini-interfaces (search results, enquiry forms, status cards), diagrams, and typographic set pieces. No photography, no real screenshots in use (eight exist unused in `public/work/`), no testimonials, no named clients — consistent with truth boundaries but leaving a proof vacuum.

**Interaction and motion.** GSAP in one client island per page; entrance reveals (from ~30–45% opacity, once, short) on effectively every section; tab crossfades; one mobile-only scroll-scrubbed journey progress; one timeline fill. Reduced motion handled properly at both the CSS layer (global kill) and JS layer (matchMedia listener + revert + `data-*-motion="reduced"`), with tests asserting no hidden content.

**Responsive character.** Genuine recomposition, not shrinkage: tab rails flip orientation (vertical list → horizontal scroll rail or grid), the systems fan survives at 400px, splits stack in a considered order, and overflow is tested at 400/360. Weaknesses: the mobile page is extremely long (~20,400px at 400w) with mostly equal-weight stacked sections, and the empty work cards become six consecutive blank screens.

## 8. Design-language assessment

**Recognisable?** Yes. Ignore the palette and the two-tone serif turn, hairline discipline, eyebrow-dash and constructed-artifact style still identify MindWP across both pages. The identity is demonstrably deeper than "Fraunces + navy + green marks" — it lives in typographic rhetoric (turns, strikethroughs, underlines, numerals) and in meaning-bearing diagram geometry.

**Ownable?** The two-tone turn + hairline + emerald-line combination is distinctive within the small-business web-services category, where competitors typically ship either SaaS-gradient or crafty-portfolio looks. The rotated fan and foundation-slab are ownable moments. Weakest ownability: the mini-interface artifacts (closest to generic SaaS illustration).

**Beautiful / premium / credible?** The rendered desktop pages are genuinely handsome; the LSEO hero is excellent. Credibility is the weak leg — not visually but evidentially: no real proof anywhere, and one section actively claims inspectability it doesn't deliver.

**Flexible?** Yes in kind: the same grammar produced a narrative homepage and a service page with different focal inventions. The industry catalogue (audiology → yacht refit) will stress it further, but nothing in the language forbids photographic or document-led pages — those lanes are simply unexplored.

**Where the weakness actually lies:**
- *Language itself:* sound. No redefinition warranted.
- *Documentation:* DESIGN.md is philosophically complete but does not name the few **shared** signatures that renders prove (two-tone turn, hairline discipline, artifact-labelling rule, axes-at-hero-scale), so each new page re-invents their mechanics (`.home-heading-muted` vs `em`).
- *CSS foundations:* one real defect (h3 role) and one missed opportunity (`--display-settings` defined but unowned by any rule).
- *Execution judgement (old era):* empty proof frames, blanket-ish reveals, repeated statement anatomy, mechanical alternation — all named by current DESIGN.md, which post-dates them. The authority already prohibits what the old page does; the page just predates the authority.

## 9. Proposed design-language evolution

The proposal is deliberately small: **preserve, name, and equip** — plus two page-proven experiments. No new visual thesis is needed.

**Evolution 1 — Name the shared signatures (documentation).**
Evidence: two-tone turn implemented divergently; axes used once; artifact labelling inconsistent between pages. Limitation: signatures that live only in page CSS get re-invented, drift, or die. Change: add a short "Recognisable shared signatures" subsection to DESIGN.md §Shared MindWP grammar naming only: the two-tone muted-turn headline (with its communication job — statement + qualification), hairline discipline, eyebrow-dash, emerald-as-connection, the constructed-artifact honesty rule (label as illustrative whenever a reasonable visitor could read it as real — as LSEO already does), and Fraunces axes as a hero-scale option. Explicitly exclude page devices (fans, slabs, numerals) from the list. Unchanged: everything else in DESIGN.md. Authority: DESIGN.md. Validation: next page build uses the shared signature without inventing a third mechanism.

**Evolution 2 — One shared implementation of the muted turn (CSS).**
Evidence: `.home-heading-muted` (span, colour + italic + weight 400, `on-dark` variant, sometimes `display:block`) vs `em` + two raw hexes on LSEO. Limitation: the single most recognisable brand device has two codebases and three colours. Change: one global role (e.g. `.display-turn` or an adopted `h1 em`/`h2 em` convention) in `typography.css` with on-dark handling; pages keep block/inline layout control. Risk: low; both pages migrate trivially. Validation: render diff on both pages, then use on the rebuilt Homepage.

**Evolution 3 — Correct the global `h3` role (CSS).**
Evidence: `typography.css` maps `h3` to italic `--display-sm` (visually smaller than `h4`); `home.css` overrides it to normal in 6 component scopes, `.home-faq__items h3` resets to `font: inherit`, and `local-seo-authority.css` resets it page-wide (`.lseo-section h3 { font-style: normal }`). Limitation: every future page will pay the same tax, and the italic default invites accidental italic headings that dilute the deliberate italic-turn device. Change: make `h3` a normal-style working heading (serif `display-sm` roman, or a sans role — decide at implementation with renders); keep the italic display style available as an explicit class for the editorial-aside/turn roles. Unchanged: h1/h2/h4 and body scale. Validation: remove the now-redundant page overrides and re-render both pages at 1640/400 expecting zero visual change.

**Evolution 4 — Proof material lane (content + design, gated on user decisions).**
Evidence: eight real work screenshots exist unused; the live Work section is empty frames under an inspectability claim; FOUNDATION demands labelled, permitted proof. Limitation: the language has no practised pattern for real evidence — the one lane that most affects credibility. Change: once the user settles permissions/attribution/anonymisation for `public/work/*`, design the Work treatment around genuinely inspectable material (real crops at legible scale, honest labels, per-item context) as part of the Homepage rebuild; if permissions fail, recompose the section to claim only what exists (e.g. process/inspectable-craft framing) rather than shipping placeholder frames. Authority: user decision → STRATEGY.md (proof posture) → page execution. Validation: rendered Work section containing zero unlabelled fabrications.

**Evolution 5 — Motion: from blanket to concentrated (execution guidance, already documented).**
Evidence: every section on both pages carries entrance reveals; DESIGN.md already rejects blanket reveal animation. Limitation: uniform reveals spend motion budget without communicating, and mildly flatten pacing. Change: none to authorities — apply the existing rule in the next build (reserve authored motion for the hero, one or two focal relationships, and state changes; let quiet sections simply be present). Validation: rendered comparison plus reduced-motion parity as already tested.

**Durable principles (guide most future work):** typographic rhetoric first; hairline discipline; navy/paper/mist as rooms with jobs; emerald only as attention/connection; constructed artifacts always honest; quiet sections are valid; recomposition (not stacking) at narrow widths.

**Reusable foundations (documentation/CSS/components):** the named signatures (Evolution 1), the shared muted-turn role (2), corrected h3 (3), existing tokens/layout/buttons/forms unchanged.

**Page-specific creative freedom (leave open):** section composition, diagram invention, focal devices (fans, slabs, numerals, staircases), local breakpoints and geometry, per-page motion choreography, surface sequencing.

**Experiments requiring rendered proof before adoption:** Fraunces axes beyond heroes; any photography/document-evidence treatment (no current page uses one); any third container or new token.

**Rejected globalisation:** the rotated card fan, foundation slab, review numerals, leaks tab anatomy, the 36/48/64rem breakpoint trio (convergent habit, fine, but not a rule), and the fit-section good/not-fit anatomy — the last already appears near-identically on both pages; a third literal copy would tip it from family resemblance into template. None of these become rules.

## 10. Design-intelligence and execution audit

The repository *does* encode designer behaviour rather than component assembly: DESIGN.md's recursive method (§Work at page scale and focal-section scale) matches the desired loop from this audit's brief almost clause for clause — provisional progression → develop focal compositions early → co-develop copy → render while adaptable → recompose weak areas → narrow-width behaviour early. The skill repeats the method compactly without duplicating doctrine, and adds the correct stopping conditions (return only the requested outcome; critique does not authorise implementation).

Checks against the specific overfocus risks: eyebrow/heading fixation is countered ("the introduction is not automatically the complete section design"); background alternation is countered ("do not alternate surfaces mechanically"); pattern-avoidance quotas are explicitly banned ("do not count surface occurrences…"); isolated-section polish is countered (fatigue and continuity sections); source-only review is countered ("source code alone cannot prove visual quality").

Two genuine gaps:

1. **The executor's raw material is under-specified, not the executor's method.** Given PAGE-PLANS.md rows as-is, even a well-behaved executor must guess what is fixed and what proof exists (§6). This, not doctrine, is the likeliest cause of a future shallow page.
2. **A worked negative example of grouping exists, but the positive economics are implicit.** DESIGN.md tells the executor a planned section is not a band; PAGE-PLANS.md then hands them 13–16 rows per page. A one-line bridge in the PAGE-PLANS preamble ("expect several rows to share one environment; 16 rows is not 16 sections") closes the gap cheaply — the Homepage's own 15-section near-literalism shows the risk is real.

No further design-intelligence documentation is recommended; the current corpus is close to the practical maximum before it becomes ceremony.

## 19. Proposed Fable 5 operating model

One model, eight postures — selected by the user's request, not by ceremony:

- **Thinker / strategist** — read FOUNDATION+STRATEGY; challenge with evidence; distinguish durable / current / page-specific / open; record nothing without an explicit user decision; end with options and a recommendation, not edits.
- **Planner** — apply PAGE-PLANNING's contract to one page; define meaning, proof, boundaries and status; never compositions.
- **Design-language adviser** — reason from DESIGN plus *fresh renders*; propose the smallest change that unlocks better work; page-prove before globalising.
- **Designer (design-and-build)** — trigger `mindwp-design-build`; load the brief + DESIGN (+ENGINEERING when coding); work recursively page↔focal-section; co-develop copy and composition; render early at broad+narrow; recompose rather than decorate; stop at implemented, verified, unpublished.
- **Critic** — renders first; report problem → why it matters → owning decision; no edits uninvited.
- **Developer** — trigger `mindwp-frontend-quality`; smallest coherent owner; verification proportionate to risk; never reinterpret strategy mid-bugfix.
- **Foundation maintainer** — touch shared docs/CSS/skills only on demonstrated systemic evidence (the h3 rule: *all* real usage fighting a default is the bar); verify with both-page no-regression renders and the structural checks.
- **In every posture** — report drift to its owning authority instead of silently reconciling; end reports with `What is done` / `What is next`.

This is a description of how the existing documents already route work; it requires no new files.
