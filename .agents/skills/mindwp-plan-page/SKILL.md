# MindWP page planning

Plan meaning and content architecture only. Stop before visual direction, layout or implementation.

## Establish the meaning boundary

1. Read the planning phase in `docs/PAGE-WORKFLOW.md`, the relevant ledger row, `docs/STRATEGY.md`, and an active `docs/pages/<page>/page.md` when present.
2. Inspect current public copy or approved references only when they help preserve brand voice, commercial depth or a known meaning boundary. Do not inspect inner visual composition for a content-only task.
3. Reconcile supplied instructions with approved evidence. Preserve fixed meaning unless the user explicitly reopens it; never fill an evidence gap by inventing an approval.

Define at the depth the page needs:

- primary audience, offer, recognised problem, honest promise and commercial job;
- service responsibility, exclusions, proof limits, qualification and primary CTA;
- central argument and narrative progression;
- section order, eyebrows and headings;
- one communication job, first-glance takeaway, essential public meaning and truth boundary, supporting reservoir, and neighbour distinction for each material section;
- detailed-copy status: fixed, adaptable or open.

Treat each section row as a **semantic landmark**, not a visual section. Record the essential public meaning and truth boundary separately from the supporting reservoir. Supporting payloads are a reservoir of meaning and truth, not public-content counts; their details may later be synthesised or remain private, but an essential claim, boundary, distinction or CTA may not disappear. Neighbour distinctions protect rhetorical progression, not separate backgrounds, objects or visual treatments. Keep named groupings only when they clarify narrative progression, never to pre-compose visual acts.

Use prose for a straightforward page and a compact relationship table when a long or difficult page needs it. Stress-test overlap, commercial completeness, truthful claims and whether the full story answers the buyer's practical questions.

## Keep visual work out

Do not propose visual territories, central carriers, silhouettes, layouts, section compositions, components, artefacts, surfaces, motion or implementation. Do not create an execution brief for design. Once meaning is approved, the exact next action is a complete desktop visual draft by `mindwp-design-page`.

If a build prompt names an already approved page, do not re-plan it. Route directly to `mindwp-design-page`.

## Record only durable meaning

When repository updates are in scope, keep `page.md` compact and update only approved meaning, copy status, truth boundaries, reference permissions and the next action. Update the ledger's **Meaning** state only when evidence supports it. Do not create prototype or experiment records.

Return the approval-ready meaning and the single next decision. End with `What is done` and `What is next`.
